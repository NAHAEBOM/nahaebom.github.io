Operational Architecture

실제 운영 가능한 자동화 시스템 구조

이 페이지는 단순한 소개가 아니라, 실제로 운영 중이거나 구축 중인 자동매매 엔진, 데이터 수집 파이프라인, 리포트 자동화, 클라우드 운영 구조를 정리한 시스템 허브입니다.

운영 시스템 보기 구조 확인

Core Focus

  • 자동매매 엔진 설계 및 운영
  • 시장 데이터 수집 및 저장 자동화
  • 리포트 생성 및 텔레그램 전송
  • 클라우드 기반 지속 운영 구조
  • 향후 API / DB / 백엔드 확장 대응

Live Systems

운영 시스템

현재 사이트의 핵심은 연구 기록이 아니라, 실제 운영 가능한 구조를 차곡차곡 쌓아가는 것입니다.

Trading Engine

자동매매 엔진

분할매수 기반 전략, 상태 저장, 예외 처리, 로그 기록, 계좌별 조건 제어를 포함한 실전형 자동매매 엔진을 설계하고 운영합니다.

  • 전략 로직 분리
  • 상태 파일 기반 복구 가능 구조
  • 예외 처리 및 로그 관리
  • 다계좌 확장 가능성 고려
Data Pipeline

데이터 수집 시스템

시장 데이터, 거시지표, 환율, 변동성 데이터 등을 주기적으로 수집하여 저장하고, 분석과 보고서 생성에 연결합니다.

  • KRX / FRED / 환율 데이터 수집
  • 정기 업데이트 자동화
  • 파일 기반 저장 구조
  • 향후 DB 연결 가능
Reporting

리포트 자동 생성

수집된 데이터를 기반으로 그래프, 요약, 시황 문장, 텔레그램 리포트를 자동 생성하는 리포트 레이어를 구성합니다.

  • 일일 리포트 자동 생성
  • 차트 이미지 출력
  • 텔레그램 메시지 연동
  • 향후 이메일 / 웹 대시보드 확장

Architecture

운영 구조

GitHub Pages는 프론트엔드 허브 역할을 담당하고, 실제 계산·수집·전송은 외부 실행 환경에서 돌아가는 구조를 전제로 설계합니다.

1. Data Sources

KRX / 환율 / 거시지표 / 증권 API / 수동 입력 데이터

→

2. Collection Layer

Python 스크립트 / 배치 작업 / 스케줄러 / API 호출

→

3. Storage Layer

JSON / CSV / Excel / Cloud Storage / 향후 DB

→

4. Analysis & Strategy

자동매매 로직 / 지표 계산 / 백테스트 / 리포트 생성

→

5. Delivery Layer

Telegram / Website / PDF / Email / Dashboard

운영 기준 핵심: GitHub Pages는 사용자에게 구조와 결과를 보여주는 프론트엔드이고, 실제 엔진은 클라우드 또는 서버 환경에서 실행되는 분리형 구조가 가장 안정적입니다.

System Modules

모듈별 설계

나중에 백엔드와 DB를 붙여도 프론트 구조를 다시 뜯지 않도록, 시스템을 모듈 단위로 나누어 설계합니다.

Strategy Engine

매수·매도 조건, 분할 로직, 리스크 제어, 상태 복구를 담당하는 핵심 엔진

  • 전략별 독립 파일 구성
  • 설정값 분리 (.json / .env / config)
  • 실행 로그와 상태 파일 분리
  • 실패 시 복구 가능한 구조 지향

Data Collector

외부 데이터 소스에서 필요한 데이터를 정기 수집하는 모듈

  • API / 웹 수집 / 파일 업데이트
  • 휴장일 / 예외 상황 처리
  • 중복 데이터 덮어쓰기 정책
  • 향후 DB 적재 가능 구조

Storage Layer

초기에는 파일 기반, 이후에는 DB 확장 가능한 저장 레이어

  • JSON: 상태 저장
  • CSV/Excel: 시계열 저장
  • Cloud Storage: 장기 보관
  • 향후 PostgreSQL / MySQL 확장

Report Engine

데이터 요약, 차트 생성, 시황 문장화, 전달 포맷 생성을 담당

  • matplotlib 기반 차트 생성
  • 일일 리포트 자동화
  • PDF / 이미지 / 텍스트 출력
  • 향후 웹 대시보드 연결 가능

Notification Layer

결과물을 텔레그램 또는 이메일 등 외부 채널로 보내는 모듈

  • 텔레그램 메시지 전송
  • 차트 이미지 전송
  • 오류 알림 전송
  • 향후 Slack / Email 연동 확장

Site Frontend

연구, 시스템 구조, 결과물, 문서, 브랜드를 보여주는 프론트 레이어

  • GitHub Pages 기반 정적 배포
  • 문서형 구조에 강함
  • 향후 API fetch 연동 가능
  • 콘텐츠 / 리포트 허브 역할

Practical Stack

실전 운영 스택

지금은 GitHub Pages를 유지하되, 실제 운영은 외부 실행 환경과 분리해서 가져가는 것이 가장 현실적입니다.

영역 현재 추천 향후 확장

Automation Python Scripts FastAPI / Flask / Worker 구조
Scheduling cron / GitHub Actions / Cloud Scheduler Airflow / Cloud Run Jobs
Storage JSON / CSV / Excel / Cloud Storage PostgreSQL / MySQL / BigQuery
Notification Telegram Email / Slack / Push
Reports PNG / TXT / HTML / PDF Dashboard / Client Portal

Expansion Roadmap

확장 로드맵

이 사이트는 정적 페이지에서 끝나는 것이 아니라, 연구 → 시스템 → 데이터 → 상품 → 브랜드로 확장되는 구조를 목표로 합니다.

Phase 1

정적 시스템 허브 완성

사이트 구조, 페이지별 목적, 시스템 소개, 연구 기록, 문의 동선을 정리합니다.

Phase 2

자동 리포트 연결

JSON / Markdown / HTML 조합으로 최신 리포트와 시스템 상태를 사이트에 반영합니다.

Phase 3

API / 백엔드 확장

FastAPI 또는 Flask를 별도 서버에 두고, GitHub Pages 프론트에서 API를 호출할 수 있게 만듭니다.

Phase 4

브랜드 / 상품화

전자책, 강의, 리서치 리포트, 자동화 시스템 문서화 및 서비스 페이지로 연결합니다.

System-first. Research-backed. Brand-ready.
